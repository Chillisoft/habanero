This tutorial provides a hands-on guide to using the Habanero architecture to create applications that interact with databases.  The tutorial steps through a demo application, first dealing with installation and setup and moving along to more advanced usage.

Our Story

Our computer parts supply business, Replace-IT, has grown beyond all expectations and the orders are raining down from all over the country.  With so much stock moving in and out of the door these days, it wasn't going to be long before our makeshift stock control system would begin to crack up.  The poor receptionist is spending hours each day logging calls from irate customers whose orders have either been wrongly fulfilled or have simply slipped through the cracks.

It's time for a customised system, and being an IT firm, we feel we have enough know-how to code our own system.  So off we go, load up the coffee machine, crank up some rock music and pull out the SQL manual for the first time.

We've decided to code the system in C# with object orientation, using the MySQL database to store the parts and sales data.  After getting "Hello, world" working and taking five minutes to celebrate the achievement, it's heads down for the real grind.

And it all grinds to a halt.  Having drawn up a plan of our object structure, we then sit down and map out the database table structure.  It suddenly dawns on us: how do you reconcile the two?  Surely there must be a simple way?  With a splatter of keyboard strokes and a tick-tick of mouse buttons, our browser calls up the Habanero architecture to solve all the world's problems ... well, except for plate tectonics.

The Problem

The microwave culture is hitting programming and why not?  The hunt is on to find an easier, quicker, cleaner and more error-free way to achieve a programming task.  Despite continued innovation, there are still some glaring efficiency gaps, including SQL's jagged interface with object-oriented programming.

Habanero was developed by a group of software developers that ran into the continuous problem of having to provide the intermediate.  A pattern of usage developed and Habanero was packaged together with recurring functionality from the different applications.  Furthermore, the architecture remained in use by the team for a long period before being released onto the market, considerably improving stability and helping to ensure that all basic functionality is included.

Revisiting the Business

Having decided on using Habanero, it's time to get this vehicle out of the garage and onto the road.  Our objective is to set up our system incrementally, getting the basic structure up and running first and then adding the bells and whistles as we go.

Designing the Database

Before we begin on the code side, let's set up the database tables.  At this stage you'll have to have chosen your database of choice and installed it as per instructions in Step 1. For this tutorial we'll be using the MySQL database.  We open the database administrator and create a new schema called "replace-it".  It's time for our first table, which will hold our computer parts.  We'll call it ComputerPart - no surprises there.  We'll start with three columns.  The first is "ID" which serves as the primary key.  Now there's quite a debate on primary keys and what format they take, but for our example, our primary key will be a Guid and will remain discrete.  In MySQL, a Guid can be represented as a CHAR(38) type.  Our other two fields will be Description as VARCHAR(500) and the Price as DECIMAL(10,2).  Here is an image of our structure as in the MySQL administrator:

Setting up the Project/Solution

Fire up your standard IDE or text editor and set up a new project.  If you're using Visual Studio, for instance, add a new project called "Repair-it" as a Windows Application (in this example).  You might also want to add a "lib" folder under "Repair-it", to store the dll's needed by the project, and a "docs" folder for any notes you want to take or documentation you want to write.

First up we'll need to reference some DLL's that are important to our application.  Unless your system is setup otherwise, you may want to put the Habanero DLL's in the "lib" folder - there should be 18 altogether.  You may also need the following DLL's:

  • log4net.dll - This is the logging system that Habanero supports for outputting runtime information to a specified log.  See Appendix I for more information.
  • Microsoft.ApplicationBlocks.ExceptionManagement.dll and Microsoft.ApplicationBlocks.ExceptionManagement.Interfaces.dll - Used for throwing exceptions.
  • MySql.Data.dll - Used to manage MySql datagase connections (if you're using a different database, you'll need to make appropriate adjustments here)

These will all need to be referenced from within your IDE as well.

Weaving in Habanero

We're going to begin by building a simple application that displays computer parts in a grid and allows us to add to and edit the list.

We'll first set up a new class for ComputerPart, seeing as the intention of Habanero is to provide objects that mirror database tables. As with all such objects, ComputerPart inherits from BusinessObjectBase.  Here is the code for a simple version of the class that provides properties to manipulate the class's fields.  Note that if you're wanting to vary this for your own use, you'll only need to change the class name where appropriate and add or remove the properties for each of the fields you want to display or amend in your database table.

ComputerPart.cs 
using System;
using System.Collections.Generic;
using System.Text;
using Chillisoft.Bo.v2;
using Chillisoft.Bo.ClassDefinition.v2;

namespace Replace-it
{

public class ComputerPart : BusinessObjectBase
{
	public ComputerPart(ClassDef classDef) : base(classDef) 
	{
		mClassDef = classDef;
	}

	public ComputerPart() : base() { }

	public static ComputerPart Create() 
	{
		return (ComputerPart)ClassDef.GetClassDefCol()[typeof(ComputerPart)].CreateNewBusinessObject();
	}

	public Guid ComputerPartID
	{
		get { return (Guid)this.GetPropertyValue("ComputerPartID"); }
		set { this.SetPropertyValue("ComputerPartID", value); }
	}

	public string Description 
	{
		get { return this.GetPropertyValueString("Description"); }
		set { this.SetPropertyValue("Description", value); }
	}

	public decimal Price
	{
		get { return (decimal)this.GetPropertyValue("Price"); }
		set { this.SetPropertyValue("Price", value); }
	}
}
}

Writing the Class Definitions

We have one last step now before our vehicle can hit the highway, and that is to provide the class definitions for our data.  We're basically specifying how the database data is to be interpreted, as well as what information we'd like to display in the grid.  We add a new ClassDefs.xml file to the project (in Visual Studio you will also need to set the properties for the xml file so that it copies to the "debug" folder).

<?xml version="1.0" encoding="utf-8" ?>
<classDefs>
  <classDef name="ComputerPart" assembly="Repair-it"  tableName="computerpart">
    <propertyDef name="ComputerPartID" type="Guid" />
    <propertyDef name="Description" />
    <propertyDef name="Price" type="Decimal" />
    <primaryKeyDef>
      <prop name="ComputerPartID" />
    </primaryKeyDef>
    <uiDef>
      <uiGridDef>
        <uiGridProperty heading="Description" propertyName="Description" />
      </uiGridDef>
      <uiFormDef height="150" width="300" heading="Add a computer part">
        <uiFormTab name="Description">
          <uiFormColumn width="250">
            <uiFormProperty label="Description:" propertyName="Description" />
            <uiFormProperty label="Price:" propertyName="Price" />
          </uiFormColumn>
        </uiFormTab>
      </uiFormDef>
    </uiDef>
  </classDef>
</classDefs>

Xml files like these also require associated DTD's and you'll need to copy the 26 Habanero's DTD's to the "debug" or output folder in your project.

Note that in this application we haven't included the ComputerPartID in the grid.  This was partly a matter of choice and partly of principle.  Regarding choice, a Guid is not as useful as a description when listing parts.  Regarding principle, it's commonly held opinion that you shouldn't be changing your primary key, so we've designed the database to eliminate that possibility, keeping the primary key discrete.  Of course, we will later add a unique reference number for each computer part, but that will be in a custom format rather than as a Guid, which is admittedly difficult to read and communicate.

In order to access the different data tables, we'll need to add a menu to the window.  We'll refactor out the common code to load the business object collection, placing it in the method SetBOCollection(), which takes a business object that will be used as a template to create a business object collection and load all the objects held in the database with that type.  To display a different data set, we simply specify a different business object collection for the ReadOnlyGrid.  We'll also add a method to create a main menu with options to view each of our data collections.  The three methods in Form1.cs now look as follows, and the application should run successfully, allowing you to navigate between the three tables and add and edit entries.

private ReadOnlyGridWithButtons grid;

public Form1()
{
    SetBounds(0,0,640,480);
    CreateMainMenu();
    InitializeComponent();

    BorderLayoutManager manager = new BorderLayoutManager(this);
    grid = new ReadOnlyGridWithButtons();
    SetBOCollection(ComputerPart.Create());
    manager.AddControl(grid, BorderLayoutManager.Position.Centre);
}

private void CreateMainMenu()
{
    MainMenu menu = new MainMenu();
    MenuItem fileMenu = new MenuItem("&Data");
    fileMenu.MenuItems.Add(new MenuItem("Computer &parts", delegate
        { SetBOCollection(ComputerPart.Create()); }));
    fileMenu.MenuItems.Add(new MenuItem("Customer &orders", delegate
        { SetBOCollection(CustomerOrder.Create()); }));
    fileMenu.MenuItems.Add(new MenuItem("&Supplier orders", delegate
        { SetBOCollection(SupplierOrder.Create()); }));

    menu.MenuItems.Add(fileMenu);
    Menu = menu;
}

private void SetBOCollection(BusinessObjectBase bo)
{
    BusinessObjectBaseCollection boCollection = new BusinessObjectBaseCollection(bo);
    boCollection.Load("", "");

    grid.SetBusinessObjectCollection(boCollection);
}
